## 一、MCP 服务问题分析（现状与目标）

以下为从当前代码与使用中归纳的**实际问题**，用于指导 MCP 服务优化与 OKR 拆解。

### 1.1 可用性不高


| 现象       | 原因/证据                                                                                        |
| -------- | -------------------------------------------------------------------------------------------- |
| 启动即降级    | `main.py` 中 MCP 初始化失败仅打 warning，图用空 `mcp_tools` 构建，analysisExpert 无 ONNX/Calphad 工具，核心能力不可用。 |
| 无健康感知    | 无 MCP 健康检查或心跳，服务宕机/网络抖动无法提前发现，只能到调用时报错。                                                      |
| 调用易超时/失败 | 架构文档（REFACTORING_PLAN）明确标注：工具层无统一重试/超时，MCP 调用易超时无保障。                                         |
| 无重试与降级   | 单次调用失败即向上抛异常，无重试、退避或降级策略（如只读缓存、跳过单任务继续执行）。                                                   |


### 1.2 工具描述不规范


| 现象            | 原因/证据                                                                                        |
| ------------- | -------------------------------------------------------------------------------------------- |
| 依赖 Agent 侧补描述 | `mcp_service.py` 注释：「MCP 原生工具描述太简短，这里提供详细描述」；用 `TOOL_DESCRIPTIONS` 字典覆盖 MCP 返回的 description。 |
| 双端维护成本高       | 参数说明、composition 计算规则、单位（K、质量%等）在 Agent 提示词与 MCP 工具描述中重复维护，MCP 侧变更易导致不一致。                    |
| 缺少机器可读规范      | 理想情况应为 MCP 服务端提供完整、稳定的 schema（参数类型、必填、单位、枚举、示例），Agent 侧仅做转发与校验，减少幻觉与调参错误。                    |


### 1.3 请求/响应数据处理与返回数据不规范


| 现象               | 原因/证据                                                                                                                         |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 返回格式以「人类可读字符串」为主 | `mcp_parser.py` 注释：「MCP 服务（ONNX、Calphad）返回的是人类可读的格式化字符串，需要解析提取其中的结构化数据」。                                                      |
| Agent 侧大量正则解析    | ONNX 结果用正则抠 UTS/YS/EL、推理时间、UUID；Calphad 结果从「计算结果:」后抠 JSON 块，多 fallback，说明实际返回格式不统一、易随版本变化。                                    |
| 解析失败只落 raw       | 解析不到关键字段时置 `parse_error`/`parse_warning` 并保留 `raw`，前端/LLM 只能看到原始字符串或二次解析，体验与稳定性差。                                             |
| 前端/会话结果依赖 JSON   | `routes.py` 对工具结果做 `json.loads(content)` 尝试，说明 ToolMessage 的 content 常为字符串而非稳定结构；前端期望结构化字段（如 UTS、YS、EL）需在 Agent 侧或 MCP 侧统一约定。 |


### 1.4 优化方向（供 MCP 服务侧落地）

- **可用性**：提供健康检查接口、超时与重试策略、明确错误码与降级策略。  
- **工具描述**：在 MCP 服务端输出完整、稳定的工具描述与参数 schema（类型、必填、单位、composition 计算规则等），减少 Agent 侧覆盖与重复维护。  
- **数据规范**：请求/响应采用稳定结构（如 JSON），统一字段命名、单位、状态码；ONNX 返回结构化性能指标，Calphad 返回统一结构的 task_id/status/result，便于 Agent 与前端直接使用、逐步弱化或移除 Agent 侧正则解析。

## 五、版本发布公告（飞书 — 软件开发部群）

---

**【版本上线公告】铝合金智能设计系统 v0.1.0**

**版本号：** v0.1.0  

**版本功能特性：**  

- **多智能体对话**：支持用自然语言描述铝合金需求，系统自动完成「需求分析 → 历史数据查询与成分推荐 → 力学性能预测 → 热力学计算 → 报告生成」全流程。  
- **核心能力**：接入 IDME 历史数据库、ONNX 性能预测模型、Calphad 热力学计算（点计算、相分数曲线、Scheil 凝固模拟），满足 Al-Si-Mg 系压铸铝合金设计场景；**MCP 服务可用时上述能力可用，后续将随 MCP 服务优化持续增强稳定性与数据规范。**  
- **人机协同**：在关键节点提供引导选择（如「是否进行热力学计算」），Calphad 任务提交前支持批量参数确认。  
- **会话与部署**：对话支持保存与恢复；提供 Docker 部署方式，便于内网/服务器一键部署。

**下个版本预告（v0.2.0 规划）：**  

- 报告导出（Word/PDF）与结构化模板优化。  
- 更多材料体系与成分范围的扩展支持。  
- **MCP 服务可用性、工具描述与请求/响应数据规范优化（与 MCP 侧协同）。**  
- 对话与工具调用的性能与稳定性优化。

- **发布流程**：v0.1.0 上线后，由产品/项目负责人在【飞书群：软件开发部】按上述公告发布；部门负责人 1～3 个工作日内汇总至【飞书群：公司全员群】。
